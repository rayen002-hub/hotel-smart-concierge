\chapter{Build, tests et validation}
\label{chap:build_tests}

\section*{Introduction}

Ce chapitre n'est pas une release Scrum. Il constitue un \textbf{récapitulatif technique} des activités de build, de test et de validation réalisées tout au long des sprints, dans le cadre de la Définition de Done définie au Sprint 0. Il présente les commandes de build pour chaque module, les résultats des tests automatisés (backend Jest/Supertest, service IA pytest) et la validation fonctionnelle globale du système LoomStay par un scénario de démonstration.

% =============================================================
\section{Rappel de la Définition de Done}
\label{sec:rappel_dod}

Conformément aux principes Scrum adoptés dans ce projet, chaque sprint a produit un incrément fonctionnel intégré, testé et validé. Les activités de build et de test ne constituent donc pas une release séparée, mais des activités continues intégrées à chaque sprint.

À chaque fin de sprint, les critères suivants ont été vérifiés :
\begin{itemize}
  \item Code compilé sans erreur (TypeScript strict, \texttt{flutter analyze}) ;
  \item Tests unitaires et d'intégration passés avec succès ;
  \item Fonctionnalité vérifiée manuellement selon le scénario de test ;
  \item Aucune régression sur les sprints précédents (suite de tests complète relancée) ;
  \item Incrément commité sur la branche principale du dépôt GitHub.
\end{itemize}

\section{Build frontend React/Vite}
\label{sec:build_frontend}

Le frontend (web-app) est construit avec Vite et TypeScript. La commande de build de production est :

\begin{lstlisting}[language=bash, caption={Build de production — Frontend React/Vite}]
cd web-app
npm run build
\end{lstlisting}

Le build génère le répertoire \texttt{dist/} contenant :
\begin{itemize}
  \item Les fichiers JavaScript minifiés et regroupés par chunk (code splitting automatique) ;
  \item Les assets CSS et images optimisés ;
  \item Le \texttt{manifest.webmanifest} pour la PWA ;
  \item Le service worker généré par \texttt{vite-plugin-pwa} pour le comportement offline.
\end{itemize}

\section{Build backend Node.js/Express}
\label{sec:build_backend}

Le backend est développé en TypeScript et compilé vers JavaScript via \texttt{tsc} :

\begin{lstlisting}[language=bash, caption={Compilation TypeScript — Backend Express}]
cd backend
npx tsc --noEmit   # Vérification de types uniquement
npx tsc            # Compilation vers dist/
\end{lstlisting}

En environnement de développement, le backend est lancé directement avec \texttt{ts-node} via \texttt{npm run dev}. En production, le serveur est lancé depuis les fichiers compilés :

\begin{lstlisting}[language=bash, caption={Démarrage du backend en production}]
node dist/index.js
\end{lstlisting}

Le backend expose un endpoint de health check \texttt{GET /api/health} retournant \texttt{\{"status":"ok"\}} pour permettre la surveillance du service.

\section{Build service IA FastAPI}
\label{sec:build_ia}

Le service IA Python/FastAPI est lancé directement avec Uvicorn :

\begin{lstlisting}[language=bash, caption={Lancement du service IA — développement}]
cd ai-service
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
\end{lstlisting}

Pour la démonstration, le service est conteneurisé via Docker et déployé sur Hugging Face Spaces :

\begin{lstlisting}[language=bash, caption={Build Docker — Service IA}]
cd ai-service
docker build -t loomstay-ai-service .
docker run -p 8000:8000 loomstay-ai-service
\end{lstlisting}

\section{Build APK Flutter}
\label{sec:build_apk}

L'application Flutter est compilée pour Android en mode release :

\begin{lstlisting}[language=bash, caption={Build APK release — Application Flutter worker}]
cd worker-app
flutter pub get
flutter build apk --release
\end{lstlisting}

L'APK release est généré dans \texttt{build/app/outputs/flutter-apk/app-release.apk}. Il est distribué directement aux employés pour installation sur leurs appareils Android.

\begin{itemize}
  \item \textbf{Version} : 1.0.0+1 ;
  \item \textbf{Taille APK} : \textasciitilde{}25 Mo (incluant les librairies Flutter et le scanner QR) ;
  \item \textbf{Cible Android} : SDK minimum 21 (Android 5.0+), SDK cible 34.
\end{itemize}

\section{Tests service IA}
\label{sec:tests_ia}

Le service IA dispose de tests automatisés couvrant les trois services principaux :

\begin{itemize}
  \item \textbf{test\_language.py} : détection de langue pour les 5 langues supportées (FR, EN, AR, ES, DE) ;
  \item \textbf{test\_classifier.py} : prédiction de catégorie pour les 6 classes (MAINTENANCE, HOUSEKEEPING, RESTAURANT, RECEPTION, NOISE, OTHER) ;
  \item \textbf{test\_translation.py} : qualité de traduction (score BLEU calculé via \texttt{sacrebleu}).
\end{itemize}

\begin{lstlisting}[language=bash, caption={Exécution des tests du service IA}]
cd ai-service
pytest tests/ -v
\end{lstlisting}

\section{Validation du scénario de démonstration}
\label{sec:validation_demo}

Le scénario de démonstration complet a été validé de bout en bout sur l'environnement de déploiement temporaire (Vercel + Render + Hugging Face + Supabase). Il couvre l'ensemble des 8 acteurs du système et les 9 sprints de développement.

La plateforme LoomStay a démontré :
\begin{itemize}
  \item La fluidité de l'expérience client via la PWA (accès QR, messagerie, réclamations, services) ;
  \item La réactivité des notifications temps réel (Socket.IO) entre les acteurs ;
  \item La précision du pipeline IA (\textasciitilde{}86\% sur les catégories testées) ;
  \item La fiabilité du scan QR dans l'application Flutter sur Android physique.
\end{itemize}
