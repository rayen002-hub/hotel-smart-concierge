\chapter{Sprint 0 : Analyse et préparation}
\label{chap:sprint0}

\section*{Introduction}

Ce chapitre constitue le Sprint 0 du projet. Il regroupe l'ensemble des activités d'analyse et de préparation préalables au développement : l'identification des acteurs, la spécification des besoins fonctionnels et non fonctionnels, la constitution du backlog produit, la planification des releases, la définition de l'architecture logique et la mise en place de l'environnement de travail. Ces activités ont été réalisées avant le démarrage du premier sprint de développement.

% =============================================================
\section{Analyse et spécification des besoins}
\label{sec:analyse_besoins}

L'analyse des besoins a été menée en collaboration avec le superviseur de l'organisme d'accueil, Loomens, afin d'identifier les fonctionnalités nécessaires à la plateforme LoomStay et les contraintes techniques à respecter. Cette phase a abouti à la formalisation des besoins fonctionnels par acteur, des besoins non fonctionnels transversaux, et à la constitution du backlog produit initial.

\section{Définition de Done}
\label{sec:definition_done}

Dans ce projet, un incrément de sprint est considéré comme terminé (\textit{Done}) lorsque les fonctionnalités développées satisfont les critères suivants :

\begin{itemize}
  \item Code développé, intégré et versionné dans la branche principale du dépôt GitHub ;
  \item Tests unitaires et/ou d'intégration écrits et passés avec succès (Jest/Supertest pour le backend, pytest pour le service IA) ;
  \item Fonctionnalité vérifiée manuellement selon le scénario de test défini en planification de sprint ;
  \item Absence d'erreurs de compilation (TypeScript, Flutter analyze) ;
  \item Code documenté (commentaires, routes API vérifiées avec Postman) ;
  \item Incrément potentiellement livrable, sans régressions sur les sprints précédents.
\end{itemize}

Le chapitre~\ref{chap:build_tests} \textit{\og~Build, tests et validation~\fg{}} est un chapitre de validation technique récapitulatif présentant les résultats cumulés des activités de test et de build sur l'ensemble des sprints. Il ne constitue pas une release Scrum.

\section{Architecture logique}
\label{sec:architecture}

\subsection{Architecture globale : SOA}

Le système LoomStay adopte une \textbf{Architecture Orientée Services (SOA)} composée de 4 modules indépendants communiquant via des API REST :

\begin{enumerate}
  \item \textbf{web-app} (React + Vite) — interfaces web : PWA installable pour l'espace client chambre (route \texttt{/room}) et dashboards pour le personnel (admin, réception, managers) ;
  \item \textbf{backend} (Node.js + Express.js) — logique métier centrale, API REST, WebSocket Socket.IO, ORM Prisma, PostgreSQL ;
  \item \textbf{ai-service} (Python + FastAPI) — intelligence artificielle : traduction multilingue (NLLB-200) et classification des réclamations (Logistic Regression + TF-IDF) ;
  \item \textbf{worker-app} (Flutter/Dart) — application mobile native Android pour les employés terrain (maintenance et ménage).
\end{enumerate}

\figplaceholder{Diagramme d'architecture globale du système LoomStay (SOA, 4 modules)}

\subsection{Architecture par composant}

Chaque module du système suit un patron architectural propre :

\begin{itemize}
  \item \textbf{Backend API} : Architecture en couches (\textit{Layered Architecture}) — \texttt{routes/} $\rightarrow$ \texttt{controllers/} $\rightarrow$ \texttt{services/} $\rightarrow$ Prisma ORM $\rightarrow$ PostgreSQL. Séparation claire des responsabilités : routing HTTP, traitement des requêtes, logique métier, accès aux données.

  \item \textbf{Frontend React} : Application mono-page (\textit{SPA}) avec React Router DOM, composants réutilisables (\texttt{AuthGuard}, layouts), gestion d'état locale (\texttt{useState}/\texttt{useEffect}). L'espace client chambre (\texttt{/room}) est configuré comme PWA installable via \texttt{vite-plugin-pwa}. Les dashboards du personnel restent des interfaces web classiques.

  \item \textbf{Flutter App} : Architecture en couches par feature (\textit{Feature-layered Architecture}), proche du Clean Architecture simplifié. Organisation : \texttt{features/\{auth,tasks,splash,messages\}/} avec sous-dossiers \texttt{data/} (models, services) et \texttt{presentation/} (screens, widgets).

  \item \textbf{Service IA} : Architecture en couches — \texttt{main.py} (routes FastAPI) $\rightarrow$ \texttt{services/} (classifier\_service, translation\_service, language\_service) $\rightarrow$ \texttt{models/} (fichiers modèle sérialisés).
\end{itemize}

\subsection{Communication et sécurité}

\begin{itemize}
  \item \textbf{REST API HTTPS} : tous les clients (web, mobile) communiquent avec le backend via des requêtes HTTP REST sécurisées ;
  \item \textbf{WebSocket (Socket.IO)} : communication bidirectionnelle en temps réel pour les notifications (nouvelles réclamations, changements de statut, messages) ;
  \item \textbf{JWT + RBAC} : JSON Web Token pour les sessions du personnel (5 rôles : ADMIN, RECEPTIONIST, MAINTENANCE\_MANAGER, HOUSEKEEPING\_MANAGER, EMPLOYEE) ; tokens signés pour les QR codes clients et workers ;
  \item \textbf{API HTTP interne} : le backend communique avec le service IA via des requêtes HTTP REST internes.
\end{itemize}
