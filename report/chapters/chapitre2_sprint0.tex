\section{Définition de Done}
\label{sec:definition_done}

Dans ce projet, un incrément de sprint est considéré comme terminé lorsque les fonctionnalités développées sont intégrées, testées, validées et documentées. Les tests, la construction de l'application et la validation ne sont donc pas traités comme une release séparée, mais comme des activités continues intégrées à chaque sprint, conformément aux principes Scrum.

La \ac{DoD} de chaque sprint comprend :
\begin{itemize}
  \item Code développé et intégré dans la branche principale ;
  \item Tests unitaires et/ou d'intégration passés avec succès ;
  \item Fonctionnalité vérifiée manuellement selon le scénario de test ;
  \item Code revu et documenté ;
  \item Incrément potentiellement livrable.
\end{itemize}

Le chapitre~\ref{chap:build_tests} \textit{<<~Build, tests et validation~>>} est un chapitre de validation technique récapitulatif, et non une release Scrum.

\section{Architecture logique}
\label{sec:architecture}

\subsection{Architecture globale : SOA}

Le système adopte une \textbf{Architecture Orientée Services (SOA)} composée de 4 modules indépendants communiquant via des API REST :

\begin{enumerate}
  \item \textbf{web-app} (React) — interfaces web (PWA client chambre + dashboards personnel) ;
  \item \textbf{backend} (Express.js API) — logique métier centrale ;
  \item \textbf{ai-service} (FastAPI) — intelligence artificielle (traduction et classification) ;
  \item \textbf{worker-app} (Flutter) — application mobile pour les employés.
\end{enumerate}
